# Oracle AI Vector Search — AI Skill Definition

## Purpose
This skill teaches AI assistants how to implement AI-powered semantic search, RAG (Retrieval-Augmented Generation) pipelines, and vector similarity operations using Oracle Database 23ai's native VECTOR type and DBMS_VECTOR / DBMS_VECTOR_CHAIN packages. Use this skill whenever storing embeddings, building similarity search, or connecting an LLM to Oracle data.

---

## VECTOR Data Type

Oracle 23ai adds a native `VECTOR` column type for storing high-dimensional floating-point arrays (embeddings). Always specify the number of dimensions and the storage format to match the embedding model's output.

```sql
-- Table with a vector embedding column (1536 dimensions = text-embedding-3-small)
CREATE TABLE IF NOT EXISTS knowledge_chunks (
    chunk_id     NUMBER          GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    doc_id       NUMBER          NOT NULL,
    chunk_text   CLOB            NOT NULL,
    embedding    VECTOR(1536, FLOAT32),    -- 1536-dim, 32-bit float
    created_at   TIMESTAMP       DEFAULT SYSTIMESTAMP NOT NULL
);

-- Supported dimension counts: any positive integer (model-dependent)
-- Supported storage formats:
--   FLOAT32  — 4 bytes per dimension; good balance of precision and size (default)
--   FLOAT64  — 8 bytes; highest precision, largest storage
--   INT8     — 1 byte; quantised, smallest footprint, slight precision loss
--   BINARY   — 1 bit per dimension; for binary hash embeddings (Hamming distance)

-- Insert a literal vector (usually generated by DBMS_VECTOR; manual insert for testing)
INSERT INTO knowledge_chunks (doc_id, chunk_text, embedding)
VALUES (1, 'Oracle 23ai introduces native vector search.', TO_VECTOR('[0.021, -0.034, 0.008]'));

-- Cast a JSON array string to VECTOR
SELECT TO_VECTOR('[0.1, 0.2, 0.3]', 3, FLOAT32) AS v FROM DUAL;

-- Retrieve vector as a string for inspection
SELECT FROM_VECTOR(embedding) AS vec_str FROM knowledge_chunks WHERE chunk_id = 1;
```

## Generating Embeddings with DBMS_VECTOR

`DBMS_VECTOR.EMBED_TEXT` calls the configured embedding model and returns a `VECTOR`. Embed content in batch using `INSERT ... SELECT` for efficiency.

```sql
-- Embed a single string (useful in PL/SQL logic or for a query vector)
DECLARE
    v_vec  VECTOR(1536, FLOAT32);
BEGIN
    v_vec := DBMS_VECTOR.EMBED_TEXT(
        credential_name => 'OPENAI_CRED',
        params          => JSON_OBJECT(
            'provider' VALUE 'openai',
            'model'    VALUE 'text-embedding-3-small'
        ),
        content         => 'What is Oracle Autonomous Database?'
    );
    -- Use v_vec for similarity search ...
END;
/

-- Batch embed: embed all un-embedded chunks in one INSERT...SELECT
INSERT INTO knowledge_chunks (doc_id, chunk_text, embedding)
SELECT
    s.doc_id,
    s.chunk_text,
    DBMS_VECTOR.EMBED_TEXT(
        credential_name => 'OPENAI_CRED',
        params          => JSON_OBJECT(
            'provider' VALUE 'openai',
            'model'    VALUE 'text-embedding-3-small'
        ),
        content => s.chunk_text
    )
FROM source_chunks s
WHERE s.embedded = 'N';
COMMIT;

-- Mark source as embedded
UPDATE source_chunks SET embedded = 'Y' WHERE embedded = 'N';
COMMIT;
```

## VECTOR_DISTANCE and Distance Metrics

`VECTOR_DISTANCE(vec1, vec2, metric)` computes the distance between two vectors. Lower values mean more similar. Choose the metric that matches what the embedding model was trained with.

```sql
-- COSINE distance: best for text embeddings (OpenAI, Cohere, most transformer models)
-- Range [0, 2] for normalized vectors; 0 = identical, 2 = opposite
SELECT chunk_id, chunk_text,
       VECTOR_DISTANCE(embedding, :query_vec, COSINE) AS cosine_dist
FROM knowledge_chunks
ORDER BY cosine_dist
FETCH FIRST 5 ROWS ONLY;

-- DOT PRODUCT (inner product): use with models trained using dot product loss
-- Higher absolute value = more similar; negate for ascending ORDER BY
SELECT chunk_id,
       VECTOR_DISTANCE(embedding, :query_vec, DOT) AS dot_dist
FROM knowledge_chunks
ORDER BY dot_dist;

-- EUCLIDEAN (L2): classic geometric distance
SELECT chunk_id,
       VECTOR_DISTANCE(embedding, :query_vec, EUCLIDEAN) AS l2_dist
FROM knowledge_chunks
ORDER BY l2_dist
FETCH FIRST 5 ROWS ONLY;

-- EUCLIDEAN_SQUARED: faster than EUCLIDEAN (avoids sqrt), same ordering
SELECT chunk_id,
       VECTOR_DISTANCE(embedding, :query_vec, EUCLIDEAN_SQUARED) AS l2sq
FROM knowledge_chunks
ORDER BY l2sq
FETCH FIRST 5 ROWS ONLY;

-- Shorthand operator notation (uses the index's default metric)
SELECT chunk_id FROM knowledge_chunks
ORDER BY embedding <=> :query_vec    -- COSINE
FETCH FIRST 5 ROWS ONLY;

SELECT chunk_id FROM knowledge_chunks
ORDER BY embedding <#> :query_vec    -- DOT (negated inner product)
FETCH FIRST 5 ROWS ONLY;

SELECT chunk_id FROM knowledge_chunks
ORDER BY embedding <-> :query_vec    -- EUCLIDEAN
FETCH FIRST 5 ROWS ONLY;
```

## Vector Indexes — HNSW and IVF

Without a vector index every similarity search performs a full table scan (exact but slow). Create a vector index for approximate nearest neighbor (ANN) search, which is orders of magnitude faster at scale.

```sql
-- HNSW (Hierarchical Navigable Small World) — best for < ~10 M vectors
-- In-memory graph structure; very low latency; higher memory cost
CREATE VECTOR INDEX IF NOT EXISTS idx_chunks_hnsw
    ON knowledge_chunks (embedding)
    ORGANIZATION INMEMORY NEIGHBOR GRAPH
    DISTANCE COSINE
    WITH TARGET ACCURACY 95;       -- 95% recall vs. exact brute-force search

-- IVF (Inverted File Flat) — scales to hundreds of millions of vectors
-- Disk-based; higher throughput; slightly lower recall at same accuracy
CREATE VECTOR INDEX IF NOT EXISTS idx_chunks_ivf
    ON knowledge_chunks (embedding)
    ORGANIZATION NEIGHBOR PARTITIONS
    DISTANCE COSINE
    WITH TARGET ACCURACY 90
    PARAMETERS (type IVF, neighbor partitions 512);

-- Tell the optimizer to use ANN search via the index
SELECT chunk_id, chunk_text
FROM knowledge_chunks
ORDER BY embedding <=> :query_vec
FETCH APPROXIMATE FIRST 10 ROWS ONLY WITH TARGET ACCURACY 95;
--      ^^^^^^^^^^ this keyword enables ANN index usage

-- Inspect vector index metadata
SELECT index_name, index_type, distance_metric, target_accuracy
FROM user_vector_indexes;

-- Rebuild index after large batch inserts (HNSW must be rebuilt in full)
DROP VECTOR INDEX IF EXISTS idx_chunks_hnsw;
CREATE VECTOR INDEX idx_chunks_hnsw ON knowledge_chunks (embedding)
    ORGANIZATION INMEMORY NEIGHBOR GRAPH DISTANCE COSINE WITH TARGET ACCURACY 95;
```

## Filtering Alongside Vector Search

Combine vector distance with standard SQL predicates. Two strategies: pre-filter (WHERE before ANN) or post-filter (ANN then WHERE).

```sql
-- Pre-filter: restrict to a subset first, then do ANN within that subset
-- Most efficient when the filter column is selective and the index is IVF
SELECT chunk_id, chunk_text,
       VECTOR_DISTANCE(embedding, :query_vec, COSINE) AS score
FROM knowledge_chunks
WHERE doc_id = :target_doc_id         -- narrow to one document first
ORDER BY score
FETCH APPROXIMATE FIRST 5 ROWS ONLY WITH TARGET ACCURACY 90;

-- Post-filter: ANN first (large candidate set), then apply business filter
-- Use when the filter column has many distinct values or low selectivity
SELECT *
FROM (
    SELECT chunk_id, chunk_text, doc_id,
           VECTOR_DISTANCE(embedding, :query_vec, COSINE) AS score
    FROM knowledge_chunks
    ORDER BY score
    FETCH APPROXIMATE FIRST 100 ROWS ONLY WITH TARGET ACCURACY 95
)
WHERE doc_id IN (SELECT doc_id FROM documents WHERE category = 'policy')
FETCH FIRST 5 ROWS ONLY;
```

## RAG Pipeline with DBMS_VECTOR_CHAIN

`DBMS_VECTOR_CHAIN` provides high-level pipeline functions: split documents into chunks, embed them, store them, retrieve semantically similar chunks, and call an LLM — all from PL/SQL.

```sql
-- Step 1: Chunk a document and embed every chunk
DECLARE
    v_doc_text  CLOB;
BEGIN
    SELECT document_text INTO v_doc_text
    FROM knowledge_docs
    WHERE doc_id = :doc_id;

    -- UTL_TO_CHUNKS splits text; UTL_TO_EMBEDDING vectorises each chunk
    INSERT INTO knowledge_chunks (doc_id, chunk_text, embedding)
    SELECT
        :doc_id,
        c.chunk_text,
        DBMS_VECTOR_CHAIN.UTL_TO_EMBEDDING(
            c.chunk_text,
            JSON_OBJECT(
                'provider'        VALUE 'openai',
                'credential_name' VALUE 'OPENAI_CRED',
                'url'             VALUE 'https://api.openai.com/v1/embeddings',
                'model'           VALUE 'text-embedding-3-small'
            )
        )
    FROM TABLE(
        DBMS_VECTOR_CHAIN.UTL_TO_CHUNKS(
            v_doc_text,
            JSON_OBJECT(
                'split'    VALUE 'words',
                'max'      VALUE 200,
                'overlap'  VALUE 20,
                'language' VALUE 'AMERICAN'
            )
        )
    ) c;
    COMMIT;
END;
/

-- Step 2: RAG query — retrieve top-k chunks then call the LLM
DECLARE
    v_user_query  VARCHAR2(2000) := 'What is the refund policy?';
    v_query_vec   VECTOR(1536, FLOAT32);
    v_context     CLOB := '';
    v_prompt      CLOB;
    v_response    CLOB;
BEGIN
    -- Embed the user question
    v_query_vec := DBMS_VECTOR_CHAIN.UTL_TO_EMBEDDING(
        v_user_query,
        JSON_OBJECT(
            'provider'        VALUE 'openai',
            'credential_name' VALUE 'OPENAI_CRED',
            'url'             VALUE 'https://api.openai.com/v1/embeddings',
            'model'           VALUE 'text-embedding-3-small'
        )
    );

    -- Retrieve top-5 most relevant chunks
    FOR rec IN (
        SELECT chunk_text
        FROM knowledge_chunks
        ORDER BY embedding <=> v_query_vec
        FETCH APPROXIMATE FIRST 5 ROWS ONLY WITH TARGET ACCURACY 90
    ) LOOP
        v_context := v_context || rec.chunk_text || CHR(10) || CHR(10);
    END LOOP;

    -- Build RAG prompt
    v_prompt := 'Use only the context below to answer the question. '  ||
                'If the answer is not in the context, say so.'         || CHR(10) ||
                'Context:' || CHR(10) || v_context                     || CHR(10) ||
                'Question: ' || v_user_query;

    -- Call LLM
    v_response := DBMS_VECTOR_CHAIN.UTL_TO_GENERATE_TEXT(
        JSON_OBJECT(
            'provider'        VALUE 'openai',
            'credential_name' VALUE 'OPENAI_CRED',
            'url'             VALUE 'https://api.openai.com/v1/chat/completions',
            'model'           VALUE 'gpt-4o-mini',
            'max_tokens'      VALUE 1000
        ),
        JSON_ARRAY(
            JSON_OBJECT('role' VALUE 'system', 'content' VALUE 'You are a helpful assistant.'),
            JSON_OBJECT('role' VALUE 'user',   'content' VALUE v_prompt)
        )
    );

    DBMS_OUTPUT.PUT_LINE(v_response);
END;
/
```

## Hybrid Search — Vector + Full-Text

For the best relevance, blend Oracle Text keyword search scores with vector similarity scores. This handles both exact keyword matches (high precision) and semantic matches (high recall).

```sql
-- Create an Oracle Text index on the chunk_text column
CREATE INDEX idx_chunks_text
    ON knowledge_chunks (chunk_text)
    INDEXTYPE IS CTXSYS.CONTEXT;

-- Hybrid search with a weighted blend
SELECT
    chunk_id,
    chunk_text,
    CONTAINS(chunk_text, :search_term)              AS text_score,
    VECTOR_DISTANCE(embedding, :query_vec, COSINE)  AS vec_dist,
    -- Normalised weighted score: 0.6 semantic + 0.4 keyword
    (
        0.6 * (1 - VECTOR_DISTANCE(embedding, :query_vec, COSINE))
      + 0.4 * (CONTAINS(chunk_text, :search_term) / 100.0)
    )                                               AS hybrid_score
FROM knowledge_chunks
WHERE CONTAINS(chunk_text, :search_term) > 0
   OR VECTOR_DISTANCE(embedding, :query_vec, COSINE) < 0.35
ORDER BY hybrid_score DESC
FETCH FIRST 10 ROWS ONLY;
```

## Best Practices

- Always specify dimension count and storage type explicitly: `VECTOR(1536, FLOAT32)`, never just `VECTOR` — the optimizer uses this for index type selection
- Use `FETCH APPROXIMATE FIRST n ROWS ONLY` (not `FETCH FIRST n ROWS ONLY`) so the optimizer can use HNSW or IVF indexes; exact search ignores the vector index
- Match the distance metric in `VECTOR_DISTANCE` to the metric specified in `CREATE VECTOR INDEX`; a mismatch forces an exact scan
- Generate embeddings in batch INSERT...SELECT, not row-by-row in a PL/SQL loop — each EMBED_TEXT call is an HTTP round-trip
- Store the model name and dimension count alongside vectors in metadata columns so you can detect if embeddings need regeneration after a model upgrade
- Set `WITH TARGET ACCURACY` to trade off recall for speed: 95% for high-quality QA, 80–90% for high-throughput recommendation workloads
- After large batch inserts into an HNSW index, drop and recreate the index — HNSW does not incrementally update efficiently
